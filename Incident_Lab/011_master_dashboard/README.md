# CyberSecurity Master Dashboard (Project 011)

The centralized orchestration hub for your cybersecurity laboratory. This tool integrates all standalone forensic, defensive, and response modules into a single, professional control panel using native OS process execution.

## Technical Explanation

* **Subprocess Orchestration:** Utilizes the `subprocess` engine to instantiate isolated child processes for each security module. This strict separation ensures that a fatal crash or memory leak in a sub-tool does not terminate the master dashboard.
* **Dynamic Parameter Injection:** Bridges interactive terminal inputs directly into standard CLI arguments (`sys.argv`), seamlessly connecting the dashboard's UI with the strict, parameter-driven architecture of the underlying forensic scripts (like passing target directories to the IR Engine).
* **Virtual Environment Persistence:** Dispatches external commands utilizing `sys.executable`, guaranteeing that all chained modules execute within the exact same Python interpreter and dependency bounds as the master orchestrator.
* **State & Signal Management:** Maintains an active, persistent loop that catches operator interrupts (`KeyboardInterrupt`) cleanly, allowing for seamless transitions between different security toolsets without dropping back to the host shell.

## Problems Solved

* **Operational Fragmentation:** Replaces multiple disjointed command-line executions with a unified, interactive interface, saving critical time for security analysts during an active incident response engagement.
* **Architecture Isolation:** Running tools as external processes (rather than directly importing their classes) shields the master dashboard from unexpected kernel-level errors generated by low-level sniffing or memory monitoring tools.
* **Professional Triage:** Provides a "ready-to-use" tactical environment, demonstrating full-stack automation, systems engineering, and architectural design skills to recruiters and peers.

## Design Decisions: "Why this instead of that?"

| Category | Decision | Why? |
| :--- | :--- | :--- |
| **Execution Engine** | `subprocess.run` | Far safer than `import`. Importing complex modules loads them into the dashboard's RAM; subprocesses keep memory isolated and automatically freed upon completion. |
| **Interface** | CLI Menu | Superior to GUI for Incident Response. It requires zero graphical dependencies, executes instantly, and functions flawlessly over remote SSH sessions. |
| **Structure** | Modular OOP | The Command Pattern allows you to easily plug in new forensic or defensive modules as your portfolio and laboratory continue to grow. |
| **Navigation** | OS-Aware `clear` | Probes the operating system (`nt` for Windows, `clear` for Unix) to provide a clean, resetting "pro-terminal" look typical of low-level security tools. |
